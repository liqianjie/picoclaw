# 游戏自动化 Skill

用于自动操作手机游戏，解放双手。

## 功能概述

- 自动打开指定的游戏应用
- 导航到指定的玩法界面
- 检测游戏状态（如菜是否成熟）
- 自动执行收菜等操作

## 触发指令

当用户说以下话语时，使用此 skill：
- "帮我收菜"
- "自动玩游戏"
- "打开[游戏名]"
- "收一下菜"
- "去[游戏名]收菜"

## 使用方法

### 1. 打开游戏应用

使用 accessibility_action 工具打开游戏：
```json
{
  "action": "launch_app",
  "package": "com.example.game",
  "target": "游戏名"
}
```

### 2. 等待游戏加载并进入主界面

获取当前屏幕状态，检查是否已进入游戏：
```json
{
  "action": "get_screen"
}
```

如果看到"进入游戏"、"开始游戏"等按钮，点击它：
```json
{
  "action": "click",
  "target": "进入游戏"
}
```

### 3. 导航到种菜/收菜界面

根据游戏界面结构，点击对应的玩法入口。可能需要：
- 点击主界面的"农场"、"种植"、"家园"等图标
- 可能需要滑动查找入口

使用 click 或 scroll：
```json
{
  "action": "click",
  "target": "农场"
}
```

### 4. 检测菜的状态

使用 screenshot_vision 或 get_screen 分析屏幕，判断菜是否成熟：
```json
{
  "action": "screenshot_vision",
  "prompt": "分析这块地里的菜是否可收获，标记出成熟菜的位置"
}
```

或者直接获取可交互元素：
```json
{
  "action": "get_screen"
}
```

### 5. 执行收菜操作

根据检测结果，点击成熟的菜：
```json
{
  "action": "click",
  "target": "收获"
}
```

如果需要精确点击位置：
```json
{
  "action": "click_position",
  "x": 500,
  "y": 800
}
```

## 完整收菜流程示例

1. 询问用户想打开哪个游戏（或从上下文推断）
2. 使用 launch_app 打开游戏
3. 等待几秒，然后 get_screen 确认界面
4. 如需登录或进入，执行 click 操作
5. 导航到种植/农场界面
6. 循环执行以下直到没有可收的菜：
   - 使用 screenshot_vision 或 get_screen 检查菜的状态
   - 点击成熟的菜进行收获
   - 如有"一键收获"按钮优先使用
7. 报告收菜结果给用户

## 调试模式

当用户说以下话语时，进入调试模式，逐步执行并输出详细信息：
- "调试模式"
- "游戏调试"
- "测试游戏自动化"
- "一步一步来"
- "帮我验证"

### 调试模式执行流程

调试模式下，每执行一步操作前，先输出以下信息：
1. **当前步骤说明**：描述即将执行的操作
2. **屏幕状态摘要**：获取到的可交互元素数量和关键元素
3. **执行计划**：具体要执行什么操作
4. **预期结果**：操作后期望看到什么变化

然后询问用户"是否继续？"，得到确认后再执行操作，并报告实际结果。

### 测试子流程

用户说"只测试打开应用"时：
1. 只执行 `launch_app` 操作
2. 获取屏幕状态并报告检测到的所有元素
3. 不执行任何点击，询问是否继续

用户说"只测试屏幕识别"时：
1. 使用 `get_screen` 获取当前屏幕
2. 详细列出检测到的所有可交互元素（按钮、文字、图标）
3. 说明可用于点击的目标名称

用户说"只测试点击"时：
1. 询问用户要点击的具体位置或目标
2. 在执行点击前明确告知坐标/目标
3. 点击后等待3秒并截图验证

## 调试指引（AI内部遵循）

在执行自动化操作时，请遵循以下调试实践：

### 1. 日志输出
每执行一步操作后，主动报告：
- ✅ 操作是否成功执行
- 📱 当前检测到的关键界面元素（前5个）
- 🎯 下一步计划执行的操作
- ⚠️ 可能遇到的问题或异常

### 2. 问题诊断
当下列情况出现时，输出诊断信息：

| 现象 | 诊断步骤 |
|------|----------|
| 无法打开应用 | 检查包名是否正确，询问用户确认游戏名称 |
| 找不到按钮 | 列出当前屏幕所有可交互元素，建议可能的替代目标 |
| 点击无反应 | 检查无障碍服务是否开启，尝试用坐标点击替代 |
| 界面识别错误 | 使用 `screenshot_vision` 获取更详细的视觉分析 |
| 循环卡住 | 检测重复操作，主动询问用户是否中止 |

### 3. 状态追踪
维护一个简化的状态变量来追踪进度：
- `step`: 当前执行到第几步（如：1-打开应用, 2-进入主界面...）
- `retry_count`: 当前步骤的重试次数（超过3次则报错并询问）
- `last_action`: 上一步执行的操作（用于检测循环）

## 注意事项

- 务必确保 PicoClaw 的无障碍服务已开启
- 游戏界面可能因版本更新而变化，需灵活调整
- 收菜操作前建议先截图确认，避免误操作
- 如遇弹窗广告，先关闭再执行主任务
- 长时间挂机注意手机电量和发热
- 部分游戏可能有检测机制，频繁自动化操作可能存在风险
- **调试模式下可以更放心地尝试，操作前会询问确认**
